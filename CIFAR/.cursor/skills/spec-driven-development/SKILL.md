# Spec-driven development (проектные спеки)

## Когда применять

Перед реализацией или при запросе пользователя: оформить задачу как спецификацию в репозитории по правилам ниже. Не смешивать спеку с кодом — только отдельный файл в `.specs/`.

## Расположение и имя файла

- **Папка**: `.specs/` в корне репозитория (создать при отсутствии).
- **Формат**: Markdown (`.md`).
- **Имя**: `YYYY-MM-DD_<slug>.md`, где `YYYY-MM-DD` — дата создания спеки (из контекста пользователя; поле «Today's date» в сессии — по умолчанию), `<slug>` — короткий идентификатор темы в `kebab-case` (латиница, цифры, дефисы).

Пример: `.specs/2026-04-26-gpu-spike-buffer.md`

## Порядок работы агента

1. Согласовать или вывести краткий `slug` из формулировки задачи.
2. Собрать недостающее: вынести в раздел **Открытые вопросы** всё, без чего нельзя зафиксировать цели или критерии приёмки.
3. Заполнить шаблон ниже; для изменений кода — явно три списка: новые / изменённые / удалённые **классы** (и при необходимости кратко другие артефакты в том же духе).
4. **Нефункциональные требования**: производительность, память, потокобезопасность, совместимость ABI/API, логирование, локализация — что уместно для задачи.
5. **Критерии приёмки**: конкретные автотесты (какие сьюиты/файлы добавить или расширить), линтеры/статический анализ (`clang-tidy`, `clang-format`, правила проекта), сборка (конфигурации), если это часть DoD.
6. Сохранить файл в `.specs/` и при необходимости указать пользователю путь.

## Шаблон спеки

Копировать в новый файл и заменить плейсхолдеры:

```markdown
# <Краткое название задачи>

- **Дата**: YYYY-MM-DD
- **Статус**: Draft | Review | Approved
- **Связанные issue/PR**: (если есть)

## Цели

Что должно быть достигнуто после выполнения задачи (измеримо, по возможности).

## Не цели

Что сознательно не делаем в рамках этой задачи (границы объёма).

## Функциональные требования

1. …
2. …

## Нефункциональные требования

- Производительность: …
- Память / ресурсы: …
- Потокобезопасность / параллелизм: …
- Совместимость (API, форматы файлов, версии toolchain): …
- Наблюдаемость (логи, метрики): …
- Прочее: …

## Открытые вопросы

Вопросы к заказчику/команде; до ответа помечать блокеры при необходимости.

1. …
2. …

## Предполагаемые изменения в коде

### Новые классы

- `Namespace::ClassName` — назначение одной строкой

### Изменённые классы

- `Namespace::ClassName` — что меняется

### Удалённые классы

- `Namespace::ClassName` — причина удаления / чем заменяется

(При отсутствии пунктов — явно писать «Нет».)

## Библиотеки и зависимости

- Новые: (имя, версия, лицензия при необходимости, зачем)
- Обновления существующих: …
- Системные / toolchain: …

## Критерии приёмки

### Автотесты

- [ ] … (конкретные тесты или сценарии)
- [ ] …

### Линтеры и качество

- [ ] … (например: профиль проекта, конкретные проверки)
- [ ] Сборка: … (Debug/Release, платформы)

## План внедрения (опционально)

Краткие этапы или порядок PR, если задача крупная.
```

## Заметки для агента

- Не выдумывать имена классов: опираться на поиск по кодовой базе; неизвестное — в **Открытые вопросы** или пометка «TBD».
- Пути в спеках указывать через `/` (не Windows-стиль в именах внутри markdown-шаблона).
- Если пользователь явно выбрал другую дату или slug — использовать их.

## Дополнительные материалы

При росте деталей вынести длинные приложения в отдельный файл рядом, например `YYYY-MM-DD_<slug>-appendix.md`, и дать одну ссылку из основной спеки.
